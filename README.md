###  2. Расчёт системных резервов Kubernetes (на основе статей)

### Формулы резервирования ресурсов

Согласно статье **"Architecting Kubernetes clusters — choosing a worker node size"**:

**Резерв CPU (формула для kubelet + ОС):**
6% от первого ядра

1% от второго ядра

0.5% от следующих двух ядер (до 4)

0.25% от всех ядер выше 4

text

**Резерв RAM (формула для kubelet + ОС):**
255 MiB для машин < 1 ГБ

25% от первых 4 ГБ

20% от следующих 4 ГБ (до 8 ГБ)

10% от следующих 8 ГБ (до 16 ГБ)

6% от следующих 112 ГБ (до 128 ГБ)

2% от памяти выше 128 ГБ

text

**Eviction threshold:** 100 МБ (из статьи)

---

#### Расчёт для выбранной конфигурации: 3 ноды × 10 ядер / 40 ГБ

##### Резерв CPU на 1 ноду:
6% от 1 ядра = 0.06 ядра (60m)

1% от 2 ядра = 0.01 ядра (10m)

0.5% от 3-4 = 0.01 ядра (10m)

0.25% от 5-10 = 0.015 ядра (15m)

Итого резерв CPU = 0.095 ядра (95m)

text

#### Резерв RAM на 1 ноду:
255 MiB (0.25 ГБ)

25% от первых 4 ГБ = 1.0 ГБ

20% от следующих 4 ГБ = 0.8 ГБ

10% от следующих 8 ГБ = 0.8 ГБ (до 16 ГБ)

6% от следующих 24 ГБ = 1.44 ГБ (до 40 ГБ)

Итого резерв RAM = 4.29 ГБ

Eviction threshold = 0.1 ГБ

Итого системный резерв = 4.39 ГБ

text

#### Ресурсы DaemonSet'ов (на каждой ноде):

| Компонент | CPU | RAM |

#### Доступные ресурсы для подов на 1 ноде:
CPU_доступно = 10 - 0.095 - 0.25 = 9.655 ядер

### 4. Дополнительные факторы из статей

#### A. Нагрузка на API Server

Согласно статье, каждый kubelet генерирует ~5 запросов в секунду к API Server:
Нагрузка на API = Количество_нод × 5 rps
Нагрузка = 3 × 5 = 15 rps

text

**Вывод:** 15 rps — допустимая нагрузка для стандартного API Server.

#### B. Загрузка container registry

При развёртывании приложений:

- Количество скачиваний образа = количество нод = **3 раза**
- При использовании 4+ нод было бы больше скачиваний
- Меньшее количество нод ускоряет развёртывание

#### C. Эффективность использования ресурсов

Из статьи:
> *"Resources reserved by the kubelet decrease with larger instances"*

Выбранные ноды (10 ядер / 40 ГБ) являются достаточно крупными, что обеспечивает:
- Минимальные потери на системные резервы (всего ~4.4% CPU и ~11% RAM)
- Низкую фрагментацию ресурсов
- Возможность размещения тяжёлых приложений (БД с 4 ГБ, кеш с 4 ГБ)
