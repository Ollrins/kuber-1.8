### 2. Расчёт системных резервов Kubernetes

#### Формулы резервирования ресурсов

**Резерв CPU (формула для kubelet + ОС):**
- 6% от первого ядра
- 1% от второго ядра
- 0.5% от следующих двух ядер (до 4)
- 0.25% от всех ядер выше 4

**Резерв RAM (формула для kubelet + ОС):**
- 255 MiB для машин < 1 ГБ
- 25% от первых 4 ГБ
- 20% от следующих 4 ГБ (до 8 ГБ)
- 10% от следующих 8 ГБ (до 16 ГБ)
- 6% от следующих 112 ГБ (до 128 ГБ)

**Eviction threshold:** 100 МБ

### 4. Дополнительные факторы из статей

#### A. Нагрузка на API Server

Каждый kubelet генерирует ~5 запросов в секунду к API Server:
Нагрузка на API = Количество_нод × 5 rps
Нагрузка = 3 × 5 = 15 rps


**Вывод:** 15 rps — допустимая нагрузка для стандартного API Server.

#### B. Загрузка container registry

При развёртывании приложений:

- Количество скачиваний образа = количество нод = **3 раза**
- При использовании 4+ нод было бы больше скачиваний
- Меньшее количество нод ускоряет развёртывание

#### C. Эффективность использования ресурсов

> *"Resources reserved by the kubelet decrease with larger instances"*

Выбранные ноды (10 ядер / 24 ГБ) являются достаточно крупными, что обеспечивает:
- Минимальные потери на системные резервы (всего ~4.4% CPU и ~14% RAM)
- Низкую фрагментацию ресурсов
- Возможность размещения тяжёлых приложений (БД с 4 ГБ, кеш с 4 ГБ)

### 7. Обоснование выбора конфигурации

#### Преимущества выбранной конфигурации (3 ноды × 10/24):

##### 1. Эффективное использование ресурсов
- Системные резервы составляют всего **~4.4% CPU** и **~14% RAM**
- Потери на DaemonSet'ы минимальны

##### 2. Достаточный запас надёжности
- При отказе одной ноды остаётся **+13.6% CPU** и **+33% RAM**
- Все реплики критичных приложений (БД, кеш) перераспределяются на оставшиеся ноды

##### 3. Оптимальная нагрузка на API
- 3 ноды генерируют всего **15 rps**
- API Server работает без перегрузок

##### 4. Быстрое развёртывание
- Всего **3 скачивания** container image
- Минимальное время старта приложений

##### 5. Гибкость масштабирования
- Возможность добавления 4-й ноды при росте нагрузки
- Запас ресурсов позволяет увеличить количество реплик на **30-50%**

##### 6. Экономическая эффективность
- 3 крупные ноды дешевле в управлении, чем 6+ мелких
- Меньше накладных расходов на мониторинг и обновления
